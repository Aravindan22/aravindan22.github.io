Async Programming

asyncio vs Celery — when to use what

  • asyncio — I/O-bound tasks in same process (HTTP calls, DB queries)
  • Celery — Background jobs, queues, retries, scheduled tasks
  • Broker — Redis (fast, simple) or RabbitMQ (robust, AMQP)
  • Never use async for CPU-bound — use multiprocessing
async def fetch_data(url): async with aiohttp.ClientSession() as s: async with s.get(url) as r: return await r.json()
Database Strategy

PostgreSQL vs MongoDB vs Redis

  • PostgreSQL — Structured, ACID, joins, default choice
  • MongoDB — Document store, dynamic schema, unstructured data
  • Redis — Cache, sessions, rate limiting, pub/sub, leaderboards
  • Read replicas — Scale read-heavy Postgres workloads
  • Connection pooling — pgBouncer for Postgres at scale
API Design Principles

REST API best practices

  • Version endpoints: /v1/users
  • Proper HTTP verbs: GET/POST/PUT/PATCH/DELETE
  • Meaningful status codes: 200/201/400/401/403/404/422/500
  • Consistent error format: {error, message, code}
  • Pagination: cursor-based for large datasets
  • Rate limiting + throttling on all public routes
  • OpenAPI spec first, then implement

Technical · Frameworks
How do you choose between Django, FastAPI, and Flask for a new project?
▾
Model Answer

It depends on the use case and team context. Django is my default for data-heavy, feature-rich enterprise apps — the ORM, admin panel, and DRF save huge amounts of time when you need CRUD-heavy backends fast.

FastAPI is my pick for high-performance async APIs, especially microservices. The auto-generated OpenAPI docs and Pydantic validation reduce errors significantly, and it handles async natively — great for I/O-heavy workloads.

Flask fits when I need minimal overhead or full control — usually for smaller services or prototypes where I want to compose my own stack.

Always mention trade-offs. Say: "Django's magic can become a footgun at scale without discipline — things like N+1 queries in the ORM need active monitoring."

▾
Model Answer (Adapt to Your Experience)

Technical · Security
How do you approach API security in enterprise projects?
▾
Model Answer

I use OWASP Top 10 as a practical checklist. Starting with authentication — OAuth2 or JWT with short-lived access tokens and refresh token rotation. Never store secrets in code — environment variables or secrets managers only.

For the data layer: parameterized queries always (never raw string formatting into SQL), input validation with Pydantic or DRF serializers, and least-privilege DB roles. At the infrastructure level — HTTPS everywhere, HSTS headers, rate limiting on all public endpoints.

I also integrate Bandit into CI pipelines for static security analysis on every PR, so issues get caught before they reach staging.

Mentioning tooling (Bandit, SAST) shows you operationalize security, not just talk about it.
05
Technical · Async
Explain asynchronous programming in Python. When would you use asyncio vs Celery?
▾
Model Answer

asyncio handles concurrency within a single process using an event loop. It's ideal for I/O-bound tasks — like making 50 external API calls concurrently — where threads would waste memory and add complexity.

Celery is for distributed background task processing — things that happen outside the request/response cycle: sending emails, generating reports, running scheduled jobs. It needs a broker (Redis or RabbitMQ) and supports retries, rate limiting, and task routing.

Important: neither helps with CPU-bound tasks. For those, I use multiprocessing or offload to a dedicated service.

A good follow-up answer: "In FastAPI I use async endpoints by default, but I still use Celery for jobs that need reliable queuing and retry semantics."

Preferred · Jinja2
How have you used Jinja2 for infrastructure or config generation?
▾
Model Answer

I've used Jinja2 to generate environment-specific config files — Nginx configurations, Kubernetes manifests, and network device configs — all from a single parameterized template.

The approach: define a base template with conditionals ({% if env == 'prod' %}), loops for repeated blocks ({% for server in upstreams %}), and filters for transformation. Render it with an environment-specific context dictionary. This means one template, three environments — dev, staging, prod — with zero copy-paste drift.

I've also implemented validation logic within templates to catch missing required variables before rendering, preventing broken configs from reaching deployment.

If you haven't used it directly, say: "I've worked with similar template-based config systems and the Jinja2 docs are clear — I'd be productive with it quickly."

Behavioral · Conflict
How do you handle a team member who consistently misses deadlines?
▾
Model Answer

I first try to understand the root cause privately — never publicly. In a 1:1, I'll ask open questions: is it unclear requirements, technical blockers, scope creep, or something personal?

Based on that, my response varies: if it's a skills gap, I pair them up or restructure their tasks. If it's unclear scope, I work with PM to tighten definitions. If it's a pattern despite support, I document it, set clear expectations, and escalate to management with evidence.

The key principle: assume good intent first, investigate before judging, and solve the system problem before the person problem.

This shows empathy + accountability — exactly what a senior leader needs to demonstrate.

Describe your experience with Docker and Kubernetes in production.
▾
Model Answer

I write multi-stage Dockerfiles to keep production images lean — build stage installs dependencies, final stage copies only the artifact. For local development, Docker Compose orchestrates the app, DB, Redis, and worker together.

In production on Kubernetes, I define Deployments with resource requests/limits, Services for internal routing, and Ingress for external traffic. I configure Horizontal Pod Autoscaler to scale on CPU/memory thresholds. For secrets, I use K8s Secrets or integrate with AWS Secrets Manager. I've also worked with Helm charts for reusable, parameterized deployments across environments.

Mention liveness and readiness probes if asked deeper — they show you understand K8s health management, not just basic deployments.

Strategy, Preparation & Execution

📅

Before the Interview

  • Review OWASP Top 10 — it's in the JD, expect a question
  • Prepare 3 project stories in STAR format
  • Practice system design: design a notification service or URL shortener
  • Review the company's tech stack / recent blog posts
  • Prepare your "tell me about yourself" — keep it under 2 mins
🧠

During — Technical Round

  • Always mention trade-offs — not just "I used X" but "I chose X over Y because..."
  • Use numbers and metrics whenever possible
  • Think out loud on system design — they want your reasoning
  • Ask clarifying questions before diving into design problems
  • It's OK to say "I haven't done exactly that, but I've done X which is similar"
🤝

During — Leadership Round

  • Use STAR format consistently for behavioral questions
  • Emphasize cross-functional collaboration — this role touches DevOps, clients, and team
  • Show empathy in conflict stories — not just "I fixed it" but how you handled the person
  • Mention specific mentoring outcomes, not just the activity
🎯

Keywords to Drop Naturally

  • OWASP, least privilege, SAST
  • 12-Factor App, strangler fig pattern
  • SLO/SLA, observability, distributed tracing
  • ADRs (Architecture Decision Records)
  • Trunk-based development, feature flags
  • DRY, SOLID, domain-driven design
🚩

Common Mistakes to Avoid

  • Don't say "we used microservices" without explaining why
  • Don't speak only technically in behavioral rounds — show leadership
  • Don't bad-mouth previous employers/clients
  • Don't over-engineer whiteboard designs — show pragmatism
  • Don't stay silent when stuck — narrate your thinking
💬

Your Opening — "Tell me about yourself"

Structure it as: Who you are → What you've built → What you're best at → Why this role excites you.

Example: "I'm a Python developer with [X] years of experience, primarily focused on building scalable backend systems. I've led teams of [N] engineers, owned architecture decisions, and worked across the full delivery lifecycle. I'm strongest in system design and mentoring. This role excites me because [specific reason]."
